# PRD: Tempest Weather Data Platform (v1)

## 1. Summary

Build a lightweight, self-hosted system that ingests data from a Tempest weather station, stores it in PostgreSQL (TimescaleDB), and provides a clean, low-friction interface for exploring current conditions and historical trends.

The system should prioritize:

* simplicity
* ownership of data + queries
* fast iteration on visualizations
* minimal operational overhead

---

## 2. Goals

### Primary goals

* Replace prior InfluxDB + Grafana setup
* Use **Postgres as the single source of truth**
* Provide **clear, useful dashboards without Grafana friction**
* Enable **ad hoc querying using SQL**

### Secondary goals

* Support future enrichment (alerts, correlations, forecasting)
* Be understandable/debuggable by one engineer
* Run reliably on a single node (NAS / small server / EC2)

---

## 3. Non-Goals (for v1)

* No mobile app
* No multi-user auth system
* No ML / forecasting
* No public sharing / API (yet)
* No complex alerting system

---

## 4. Users

### Primary user

* You (technical, SQL-capable, annoyed by opaque tools)

### Secondary (future)

* Household members (read-only dashboard)

---

## 5. System Overview

```id="sys1"
Tempest (UDP or API)
        ↓
Ingest Service (listener)
        ↓
Postgres (TimescaleDB)
        ↓
Query Layer (SQL / views)
        ↓
UI (custom dashboard or Superset)
```

---

## 6. Functional Requirements

## 6.1 Ingestion

### Requirements

* Support **UDP broadcast ingestion** (preferred)
* Optional fallback: REST polling from Tempest API
* Parse:

  * observations
  * rapid wind
  * rain events
  * lightning strikes

### Behavior

* Handle duplicate packets idempotently
* Accept slightly out-of-order data
* Log malformed packets without crashing

---

## 6.2 Storage

### Requirements

* Use PostgreSQL with TimescaleDB extension
* Store raw observations in a hypertable

### Schema (v1)

```sql
observations (
  time timestamptz NOT NULL,
  station_id text,
  air_temperature double precision,
  relative_humidity double precision,
  barometric_pressure double precision,
  wind_avg double precision,
  wind_gust double precision,
  wind_lull double precision,
  wind_direction double precision,
  rain_increment double precision,
  uv double precision,
  solar_radiation double precision,
  lightning_strike_count integer,
  lightning_avg_distance double precision,
  battery double precision
)
```

### Requirements

* Hypertable partitioned on `time`
* Index on `(station_id, time DESC)`
* Retention policy (configurable, default: keep forever)

---

## 6.3 Derived Data (Critical)

System must provide precomputed or queryable derived metrics:

### Required views

* **Hourly rain totals**
* **Daily rain totals**
* **Wind summary (avg + gust per interval)**
* **Temperature + humidity combined view**
* **Pressure trend (delta over time)**

Example:

```sql
CREATE VIEW hourly_rain AS
SELECT
  time_bucket('1 hour', time) AS bucket,
  sum(rain_increment) AS rain_total
FROM observations
GROUP BY bucket;
```

---

## 6.4 Query Layer

### Requirements

* All dashboards must use **plain SQL**
* No DSL (no Flux, no Grafana transformations)
* Queries should be readable and version-controlled

## 7. Non-Functional Requirements

### Performance

* Handle 1–10 observations per second easily
* Queries <500ms for typical dashboards

### Reliability

* Ingest service should auto-restart
* No data loss on restart

### Operability

* Single command deploy (Docker Compose preferred)
* Logs accessible locally

* Move to **custom UI once queries stabilize**
